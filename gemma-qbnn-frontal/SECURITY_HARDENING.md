# Gemma + QBNN セキュリティ強化ドキュメント

## 概要

v1.1.1 では、QBNN判断層のセキュリティを大幅に強化しました。悪意のある要求（ハッキング、マルウェア開発など）を**入力段階で検出・拒否**する仕組みを導入しました。

## 実施したセキュリティ強化

### 1. セキュリティフィルタの実装（最重要）

**ファイル**: `src/engine.ts`

#### 機能
- 入力テキストをセキュリティ関連キーワードで検査
- 検出時は即座に「No」判断を返却（QBNNを迂回）
- スコア 0点、信頼度 100% での拒否

#### 検出キーワード
```
ハッキング, 侵入, クラッキング, バックドア, ウイルス, ランサムウェア,
マルウェア, フィッシング, パスワード盗難, DDoS, 分散型サービス妨害,
不正アクセス, システム侵害, データ盗難, 暗号資産盗難, 詐欺サイト, 脆弱性悪用
```

#### 実装
```typescript
private validateSecurityRequest(userInput: string): { safe: boolean; reason?: string }
```

### 2. 入力値検証

**ファイル**: `src/engine.ts`, `src/qbnn.ts`, `src/generator.ts`

#### 実装内容
| パラメータ | 検証内容 | 制限値 |
|----------|--------|------|
| `userInput` | 入力長チェック | 最大10,000文字 |
| `numVariations` (generateBatch) | バッチサイズチェック | 最大100 |
| `entangle_strength` (QBNN) | 値域チェック | 0.0 - 1.0 |
| `seed` | 有限性チェック | 有限数のみ |
| `conversationHistory` | サイズ制限 | 最大1,000メッセージ |

### 3. エラーハンドリング

- 不正なパラメータでは明確なエラーメッセージを返却
- スタックトレースは非表示化

```typescript
if (numVariations > this.MAX_BATCH_SIZE) {
  throw new Error(`バッチサイズが大きすぎます。最大${this.MAX_BATCH_SIZE}までです。`);
}
```

### 4. メモリ安全性

**会話履歴のサイズ制限** (`src/generator.ts`)
```typescript
private readonly MAX_HISTORY_SIZE = 1000;

if (this.conversationHistory.length > this.MAX_HISTORY_SIZE) {
  this.conversationHistory = this.conversationHistory.slice(-this.MAX_HISTORY_SIZE);
}
```

## 処理パイプラインの変更

### 修正前
```
入力 → Gemma言語理解 → Gemma課題発見 → QBNN課題判断 → Gemma言語生成 → 出力
```

### 修正後
```
入力 → 【セキュリティフィルタ】→ Gemma言語理解 → Gemma課題発見 → QBNN課題判断 → Gemma言語生成 → 出力
        ↓ (検出時は即座に拒否)
      No (スコア0, 信頼度100%)
```

## 今後の改善提案

### 短期（次のマイナーリリース）
1. **ダイナミックフィルタの追加**
   - キーワード以外の悪意パターン検出（例：文脈からの判断）
   - パターンベースのシグネチャ検出

2. **ログ・監視機能**
   - セキュリティフィルタ検出ログ
   - 異常アクセスパターンの記録

3. **多言語対応**
   - 英語などの他言語の悪意キーワード検出

### 中期（メジャーバージョン）
1. **AIセーフティフレームワークとの統合**
   - Anthropic Constitutionalなどとの連携
   - より高度な判断能力

2. **ユーザー認証・認可の実装**
   - ユーザーロールごとのアクセス制御
   - リクエストの監査ログ

3. **段階的なフィルタリング**
   - グレーゾーンのリクエストはQBNNで判断
   - 明確に悪意があるもののみフィルタ
